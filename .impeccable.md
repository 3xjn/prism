## Design Context

### Users
Prism serves Roblox TypeScript/React developers building reusable UI for Roblox experiences and Studio-driven development workflows. The primary job is to provide predictable, production-minded primitives and components that feel familiar to Mantine/MUI users while respecting Roblox constraints around instances, sizing, input, portals, and composition.

Design work should optimize for developer confidence, fast component evaluation in ui-labs/Studio, and clean implementation handoff through semantic tokens, typed props, slot escape hatches, and smoke-testable behavior. Core use cases include in-game panels, settings, forms, overlays, command menus, HUD-adjacent surfaces, and developer-facing playground stories.

### Brand Personality
Prism should feel **clean, professional, and calm**.

The interface should evoke confidence and clarity rather than spectacle. It should read as a polished developer-tool design system: restrained, trustworthy, and easy to scan, with enough Roblox-native pragmatism to work well in game UI contexts.

### Aesthetic Direction
Prism should lean Mantine/MUI-like in ergonomics and visual restraint, while remaining Roblox-native in implementation and interaction details. The current blue/gray palette is the brand foundation, not merely placeholder styling: blue provides the primary accent and gray/neutral surfaces carry the calm developer-tool baseline.

The visual language should favor soft rounded surfaces, light borders, semantic-tinted states, BuilderSans typography, practical spacing, and purposeful motion. Avoid noisy game-skin aesthetics, heavy gradients, skeuomorphic panels, overanimated interactions, and arbitrary one-off colors. Components should look at home in both Studio playgrounds and polished Roblox experience UI.

Accessibility expectations are practical and ongoing: maintain readable contrast, visible state changes, keyboard/gamepad-friendly interaction paths where Roblox patterns allow, text sizes that remain legible in common game UI contexts, and reduced-motion-friendly behavior by honoring tokenized/short animations instead of relying on large movement.

### Design Principles
1. **Token-first, never ad hoc** — use semantic color, spacing, radius, typography, shadow, and motion tokens before raw values; raw Roblox units are explicit escape hatches.
2. **Calm clarity over visual noise** — prefer clean surfaces, clear hierarchy, restrained accents, and readable controls over decorative complexity.
3. **Roblox-native predictability** — components should map clearly to Roblox instances, sizing rules, input behavior, portals, and Studio smoke-test workflows.
4. **Composable but owned** — each component owns its structure and motion, while `slotProps` provide deliberate last-write-wins customization for advanced users.
5. **Interaction states must communicate** — hover, press, disabled, focus/open, destructive, and selected states should be visually distinct, accessible, and semantically colored without becoming loud.
